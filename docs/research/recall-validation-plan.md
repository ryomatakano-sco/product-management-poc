# JAN ルックアップ再現率 検証計画（recall validation plan）

**Status:** 計画のみ — 実装・API 作業は PM 合意まで着手しない
**Owner:** TBD（PM + 開発 1 名）
**所要:** 準備 0.5 日 / 実行 0.5 日 / 分析 0.5 日

## 背景と仮説

本 PoC の中核価値は「JAN を入れると AI が商品情報を自動入力する」体験にある。
現在この機能のヒット率は **未計測の想定値（~50–70%）** に依存している。
この一点が外れると登録時短の価値提案全体が崩れるため、実クリニックの
商品マスタで再現率を測るのが最優先の検証である。

**検証する仮説:** 実在する歯科クリニックの取扱商品のうち、JAN または商品名
ルックアップで有用な商品情報（名称・種別・メーカー・参考価格のいずれか2つ以上）
が得られる割合は 60% 以上である。

## プロトコル

### 1. データ収集（マスキング必須）

- 協力クリニック 1〜2 院から商品マスタの CSV エクスポートをもらう
  （必要列: 商品名 / JAN(あれば) / 種別(物販・消耗品) / メーカー名）。
- 患者・価格・仕入条件など機微情報は**受領前に先方で列削除**してもらう。
- 商品数の目安: 1 院あたり 150〜400 SKU。B2C 物販と B2B 消耗品の両方を含むこと。

### 2. 実行

- 全行を既存のルックアップ（`POST /ai/lookup` 相当）に機械的に通す
  スクリプトを書く（レート制限・コスト上限を先に決める — 1 回の実行で
  API コストが発生する点に注意）。
- JAN がある行: JAN ルックアップ → 失敗時に商品名ルックアップ。
- JAN がない行: 商品名ルックアップのみ。
- 生レスポンスを全件保存（後から採点基準を変えられるように）。

### 3. 採点（フィールド精度）

各行を人間が 15 秒以内で判定できるルーブリック:

| 判定 | 基準 |
|---|---|
| **HIT** | 名称・種別・メーカー・参考価格のうち 2 つ以上が「そのまま使える」 |
| **PARTIAL** | 1 つだけ使える / 軽微な修正で使える |
| **MISS** | 何も返らない・別商品・使いものにならない |

### 4. 集計の切り口（必須ブレークダウン)

- **B2C 物販**（歯ブラシ・ガム等、JAN 流通が厚い）vs **B2B 消耗品**
  （グローブ・印象材等、JAN-DB 収載が薄い想定）— この 2 群の差が
  プロダクト判断の核心。
- JAN あり vs JAN なし（名前ルックアップのみ）。
- メーカー別の偏り（特定メーカーだけ極端に弱い等）。

## 判断基準（事前に固定する）

| 結果 | アクション |
|---|---|
| 全体 HIT ≥ 60% かつ B2B ≥ 40% | 現行 UX のまま推進。no-result 導線の改善のみ |
| B2C は高いが B2B < 40% | 種別で UX を分岐（B2B はテンプレ/履歴ベース入力を主役に） |
| 全体 < 40% | 自動入力を「補助」に降格し、価値提案を再検討 |

## 明示的にやらないこと

- **JAN-DB 事前検索レイヤや楽天/Amazon 商品 API の実装はしない。**
  ToS 制約（検索結果の保存・商用利用制限）と PM 合意が必要なため、
  本計画の結果を見てから **推奨事項として別途起案**する。
