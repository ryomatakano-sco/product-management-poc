# ADR 0001: JAN ルックアップの再現率改善アーキテクチャ

**Status:** Proposed — **PM 承認待ち。本 ADR の選択肢はいずれも未実装。**
**Date:** 2026-07-16
**Deciders:** PM + 開発

## Context（背景）

本 PoC の中核価値は「JAN / 商品名を入れると AI が商品情報を自動入力する」体験にある。現在の実装は **モデルの Web 検索のみ**に依存しており（`services/ai_agent.py`: search agent → extraction agent）、次の構造的な天井がある。

- 再現率は体感 ~50–70%（**未計測** — 計測計画は `docs/research/recall-validation-plan.md`。A5 で入れた `ai_corrections` テレメトリが日常利用から実測値を蓄積し始める）。
- 底が抜けるのは **日本語 Web にほぼ露出しない商品** = 歯科の B2B 消耗材（グローブ・印象材・ボンディング材など）。これこそが実在庫の主役であり、Web 検索の改善プロンプトでは根本的に届かない。
- 既に入れた対策（strict-citation、wrong-product guard、title guard A4、マスターリスト注入）は**精度**（間違った商品を出さない）には効くが、**再現率**（そもそも見つかる率）は上げない。

## Options（選択肢）

### (a) JAN データベース事前検索レイヤ

Web 検索の前に JAN → 商品名/メーカーの静的 DB（JANコード統合商品情報データベース（JICFS/IFDB）、GS1 Japan データ、公開 JAN 検索サイト）を引く。ヒットすれば商品名を種にして Web 検索の的中率も上がる（JAN 単独検索より名前検索の方が強い）。

- **効果:** B2C 物販で高い（JICFS は消費財中心）。B2B 歯科材は JICFS 収載が薄く、限定的。
- **コスト/制約:** JICFS/IFDB は**有償ライセンス**（GS1 Japan 経由、利用契約が必要）。無償の JAN 検索サイトのスクレイピングは ToS 違反リスクが高く**不採用**。ローカルキャッシュテーブル（jan → name/maker）を自前で育てる形なら、A5 テレメトリ + 手入力データから **合法かつ無償で漸進的に構築できる**。
- **実装規模:** 小（lookup 前段に 1 テーブル参照を挟むだけ。`run_product_lookup` の先頭に 1 フック）。

### (b) 仕入先 PDF/CSV カタログ取り込み

クリニックが実際に発注する **ディーラーのカタログ/価格表**（PDF・CSV・Excel）を取り込み、店舗ローカルの商品マスタ辞書を作る。ルックアップはまずこの辞書を引き、ヒットしなければ Web 検索へ。

- **効果:** **B2B 消耗材に対して唯一まともに効く選択肢。** 実際に買う物はディーラーのカタログには必ず載っている。JAN が無い商品（型番のみ）にも対応できる。
- **コスト/制約:** 取り込みパイプライン（PDF テーブル抽出は LLM で行える — 既存の extraction agent 資産を流用）と、カタログ入手の運用（クリニックがディーラーから受け取る前提）。外部 API 依存なし、ToS 問題なし。
- **実装規模:** 中（アップロード UI + 抽出ジョブ + 辞書テーブル + lookup 前段フック）。

### (c) Yahoo!ショッピング商品検索 API

商品コード（JAN）検索を公式 API で行う。**Rakuten・Amazon の商品 API は本用途では ToS 上不可**（検索結果の保存・自社 DB への取り込み制限、アフィリエイト目的限定条項）— この 2 つは**選択肢から除外**する。Yahoo!ショッピング API も保存・表示条件の確認が必要（クレジット表記義務、キャッシュ期間制限あり）。

- **効果:** B2C 物販の JAN ヒット率を大きく引き上げる。B2B には効かない。
- **コスト/制約:** ToS 精査が必須（レスポンスの永続化可否が核心）。API キー管理、レート制限。**PM とビジネス側の承認なしに着手しない。**
- **実装規模:** 小〜中。

## Decision（提案 — PM 承認待ち）

**段階案:**

1. **今すぐ（承認不要・実装済み分):** A5 テレメトリで実測を開始し、`docs/research/recall-validation-plan.md` の検証を 1 クリニックの実マスタで実施する。**このデータが出るまで (a)/(c) の投資判断はしない。**
2. **第 1 投資 = (b) 仕入先カタログ取り込み。** B2C/B2B 分割計測で B2B が予想通り低ければ、(b) だけが B2B の底を上げられる。ToS 制約ゼロ・既存 LLM 抽出資産の流用で PoC 価値の実証に最も直結する。
3. **第 2 投資 = (a) の自家製キャッシュ版**（ヒットした JAN→名称を保存して次回の前段に使う。実装極小・無償）。JICFS 有償契約は商用化判断とセットで別途。
4. **(c) Yahoo API は保留。** B2C 再現率が計測で 70% を下回り、かつ ToS 精査で永続化が許容と確認できた場合のみ起案。

## Measurement plan（計測計画・事前登録）

- 手順・マスキング・採点ルーブリックは `docs/research/recall-validation-plan.md` §1–4 に従う（B2C/B2B 分割は必須）。
- 補助データ: `GET /dev/ai-corrections` の accept-rate（フィールド別）を週次で記録。
- **Go/No-Go（事前登録・変更禁止）:**
  - 全体 HIT ≥ 60% かつ B2B ≥ 40% → 現行アーキテクチャ継続、(b) は次フェーズ。
  - B2B < 40%（予想シナリオ） → **(b) を実装**（スペックレットを起票して見積り）。
  - 全体 < 40% → 自動入力を補助機能に降格し、価値提案の再設計を PM と行う。

## Consequences（帰結）

- (b) を選ぶと「AI が Web から見つける」から「店の調達データを構造化する」へ価値の重心が移る — デモの物語も「あなたのディーラーのカタログを飲み込む」に変わる（営業上はむしろ強い）。
- 前段辞書を挟むと lookup の分岐が増える — `run_product_lookup` の前段フック 1 点に閉じ込めること（C6 の分割後は `ai/lookup.py` が挿入点）。
- スケジューラ同様、取り込みジョブは単一ワーカー前提の in-process で始めてよいが、商用化時は外部ジョブランナーへ（`docs/production-readiness.md` 参照）。

## 実装しない宣言

本 ADR の時点で (a)(b)(c) いずれの**コードも書かない**。次のアクションは PM の承認と、recall-validation の実測のみ。
